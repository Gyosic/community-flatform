# 기획 및 요구사항

## 프로젝트 목표

이 플랫폼은 다음을 지원해야 합니다:

- 간단한 설치 프로세스 (Docker 또는 전통적 배포)
- 초기 설정 마법사를 통한 기본 구성
- 관리자 패널을 통한 완전한 커스터마이징
- 플러그인 시스템을 통한 기능 확장
- 백업/복원 및 마이그레이션 도구
- 성능 모니터링 및 최적화 옵션

## 핵심 기능

### 1. 관리기능 (관리자 기능)
#### 환경설정
- 사이트명, 사이트설명, 사이트로고, 파비콘, 테마, 권한, 기능, 검색엔진 등 사이트 환경 범위에 해당하는 설정을 할 수 있다.
#### 게시판설정
- 실질적으로 라우팅될 페이지 설정. 
- 라우팅 경로는 카테고리에 따라 미리 지정.
- 게시판에 필요한 설정.
- 페이지에서 보일 컴포넌트 Drag and Drop 형식으로 추가, 삭제, 크기 및 위치 변경 가능해야함.
- react-grid-layout 기반으로 설정
#### 메뉴설정
- 만들어진 게시판으로 메뉴를 설정
- 메뉴는 트리구조
- 메뉴명, 메뉴 아이콘, 라우트 경로(게시판에서 설정되어있을것임), 숨김 등 설정이 가능해야함
- 메뉴 권한 설정. 일반 회원 레벨 xx 이상부터 접근 가능 등 설정할 수 있어야 함
#### 역할관리 (권한 설정)
- 역할: 시스템관리자, 관리자, 운영진, 일반회원, 신규회원
- 게시판별 권한: 읽기/쓰기/댓글/공지
- 레벨업 기준 설정 (글 수, 댓글 수 등)
#### 사용자관리
- 사용자 개인정보 이외의 정보 설정이 가능해야함.
- 개인정보 이외의 정보 조회 가능해야함. 신고 몇번 당했고, 계정이 활성상태인지 비활성 상태인지, 언제 로그인을 했는지 등등
#### 테마관리(후순위)
- 주 색상, 보조색상 등등 색상을 설정할 수 있어야함 (Tailwind 연계)

### 2. 게시판 시스템 (사용자 기능)
- 게시글 조회
- 게시판 생성/삭제 (권한자에 한해서)

## 아키텍처 원칙

### 설정 기반 설계 (Configuration-Driven)

모든 기능은 하드코딩이 아닌 설정으로 구동됩니다. 관리자는 코드가 아닌 설정을 통해 동작을 제어합니다.

### 단일 인스턴스 아키텍처

- 각 배포는 하나의 커뮤니티만 운영
- 데이터베이스는 해당 인스턴스의 커뮤니티 전용
- 설정은 환경변수 + 데이터베이스에 저장
- 향후 확장성을 위해 설계는 유연하게 유지

### 역할 기반 접근 제어 (RBAC)

- 세분화된 권한: Role + Permission 시스템
- 게시판별 권한: 읽기/쓰기/댓글/공지
- 레벨업 조건 설정 가능 (글 수, 댓글 수 등)

## 데이터베이스 설계

```typescript
// 주요 엔티티는 다음과 같은 구조:
{
  id: string;
  created_at: timestamp;
```

### 데이터 분리

- PostgreSQL: 진실의 원천(Source of Truth), 관계 무결성
- Redis: 세션 관리, 캐싱, 실시간 알림

## 개발 가이드라인

### 기능 추가 시 필수 질문

1. **확장 포인트**: 기존 코드 수정 없이 어떻게 확장 가능한가?
2. **관리자 제어**: 관리자가 UI에서 어떻게 설정하는가?
3. **설정 분리**: 하드코딩이 아닌 설정으로 제어 가능한가?
4. **플러그인 가능성**: 향후 플러그인으로 전환 가능한가?

### 코드 구성 전략

기능 구현 시 다음 순서를 따릅니다:

1. **설계**: 확장 포인트와 설정 스키마 정의
2. **데이터 모델**: JSONB 설정을 활용한 유연한 PostgreSQL 스키마
3. **API**: 권한 검사를 포함한 안전한 엔드포인트
4. **UI**: 설정 기반 컴포넌트 + 관리자 컨트롤
5. **확장성**: 플러그인 훅과 오버라이드 메커니즘
